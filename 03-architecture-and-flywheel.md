# 03 · 系統架構與飛輪

加入 PRISMA 後,整個東西從「MTB 報告產生器」升級成「有系統性回顧當骨幹的臨床證據系統」。

## 核心重構:慢餵養快

兩條工作流跑在完全不同的節奏上,漂亮的架構是讓慢的餵養快的,而不是讓它們競爭。

MTB 報告要**快**(主治要的是「這病人現在怎麼辦」,以分鐘計);PRISMA 回顧要**慢**(多人雙篩、納入排除、可能 meta-analysis,以天/週計)。做法:病人的分子發現若對應到院內**已完成**的系統性回顧,報告直接引用那份人工驗證過的回顧(最高信任、即時);沒有對應時,agent 跑一個輕量「快速回顧」(一樣分階段、可稽核),並明確標記為未經正式回顧。

**鐵律:MTB 報告永遠先出快速版、可簽核。系統性回顧是背景非同步升級證據,不是報告的前置關卡。**

## 一個證據庫,兩個介面

MTB worklist(逐病人)與 Reviews(回顧專案)是兩種物件、兩種生命週期,但**共用同一個 Evidence Library**——依 topic / variant / drug 標籤的納入研究。對一間開不起大型 MTB、也養不起研究團隊的小醫院,「兩條線共用基礎設施」正是最強的效率論點。

## 分級的 provenance

證據背書因此有了深度:

- **完成的系統性回顧背書** → 最強;點開見完整 PRISMA flow + 納入清單。
- **快速回顧背書** → 中等;附「尚未正式回顧」caveat + 一鍵「升級為系統性回顧」。

剛好對應主治本來就懂的證據等級。

## 飛輪(全案最強差異化)

```
臨床報告暴露證據缺口(某建議只有快速回顧、低證據等級)
        ↓
缺口自動成為候選的系統性回顧題目
        ↓
團隊排進 PRISMA pipeline,完成人工雙篩驗證
        ↓
未來報告自動升級到最高信任等級
        ↓
（回到臨床現場,暴露下一個缺口)
```

**臨床現場的需求直接驅動研究議程。** 對研究機構評審,這個系統級洞察遠比「又一個臨床 AI」動人。而因為 PRISMA 已是現成的,這個迴圈可以**當場演給評審看,不是用嘴講**。

## 篩選本身也是 PR 型流程

title/abstract 篩選、full-text 篩選、雙人衝突解決,本來就是「多人審閱、include/exclude 投票、衝突像 merge conflict 浮現、資深仲裁」。所以報告審閱與回顧篩選共用同一套互動模型,學一次就會兩種。(此模型僅存在於後端心智,檯面上不外露,見 `04`。)
