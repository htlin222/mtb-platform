# 02 · 產品概念與可行性

## 概念

給小型醫院的虛擬分子腫瘤委員會。輸入病人定序結果 + 病歷號,輸出一份給主治醫師的、可簽核的 MTB 報告。整合四塊現成模組,再加上一條 PRISMA 系統性回顧作為證據骨幹。

四塊模組:

1. **NGS 三級分析** — variant → clinical annotation(分子輸入)
2. **Agentic 臨床資料獲取** — 病人脈絡:診斷、分期、前線治療、共病
3. **NCCN 指引分析** — standard-of-care 對照
4. **文獻分析(PubMed + Scopus)** — 證據合成

加上骨幹:**PRISMA 系統性回顧**工作流(見 `03`)。

## 可行性判斷:有搞頭

這是少數「本來就該長在一起」的組合,不是硬湊。四塊剛好對應真實 MTB 的完整流程:分子輸入 → 病人脈絡 → 指引對照 → 證據合成 → 結論報告。痛點是真的:MTB 極吃跨科專家與時間,中小型醫院根本開不起來。把其中幾步自動化/放大,價值明確;而多子代理協調又正是 Claude 的強項,契合比賽主題。

## 該老實面對的風險(討論中已釐清)

**臨床資料獲取是最難、風險最高的一塊。** 難的不是 AI,是 EMR 整合 + 個資/PHI + IRB。真實部署常卡在這、不在模型。**Demo 階段一律用合成/去識別化資料**,別掉進 EMR 整合的坑。

**報告讀者已定案:主治醫師,不是病人。** 這是關鍵定位——面向醫師是臨床決策輔助(human-in-the-loop),監管與責任風險遠低於直接對病人發治療建議。介面因此可以是密集、技術性的臨床文體。

**高風險領域的幻覺問題。** LLM 會捏造 variant–drug 關聯、誤讀指引、引用不存在的文獻——在腫瘤治療會出事。所以 grounding、逐條 citation、不確定性標記不是加分項,是產品核心。

**差異化。** 這塊已有 OncoKB、CIViC、Tempus、Foundation。我們的新意在於「用 agent 動態把臨床脈絡 + 指引 + 即時文獻串起來推理」這一層,加上 PRISMA 飛輪——必須講清楚,否則易被看成既有工具的包裝。

**Scopus 需機構訂閱;PubMed 免費開放(E-utilities)。** Demo 先靠 PubMed 會順很多。
