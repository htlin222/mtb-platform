# 04 · UI 設計與非技術醫師的上手

## 設計系統:Primer

採 Primer(GitHub 的設計系統)。契合原因:它本來就是給工程師「快速掃描高密度結構化資料 + 審閱 + 簽核」設計的,而 VCF 本質就是 code-like(gene、HGVS、coordinate、VAF),用等寬字呈現天然就對。VCF + 病歷號為啟動輸入。

## 最重要的一個決定:報告是 Pull Request,不是 PDF

「AI 產生草稿 → 主治審閱、留註解、覆寫建議 → 簽核放行」這條流程,就是 PR review flow。套上去免費拿到:draft 狀態、逐條建議可審、inline 註解、必要檢查(checks)、以及 approve/merge = 簽核。**但——這套模型只能留在後端心智,檯面上一個字都不能露。**

## 對非技術醫師的鐵律:PR 在水管裡,檯面只講臨床語言

再嚴謹的後端,只要 surface 洩漏一個工程/資訊科學的詞,不會寫程式的主治就退場。介面裡只有**病人、報告、依據、審閱、簽核、會診**——沒有 VCF、parse、pipeline、merge、diff、conflict。用使用者掌握、認得的詞命名,不用系統實作命名。連錯誤訊息都要人話:「這個檔案讀不到,請確認是定序報告」,而非任何技術碼。

## 招牌互動:三層閱讀深度

醫師 90% 時間只想看「這病人現在怎麼辦」,所以報告預設只給結論。

- **第一層(一眼,~10 秒)**:臨床結論——可用藥發現 + 建議,像一張他們本來就信的會診/病理報告文體。醫師日常停留處。
- **第二層(掃讀,~1 分)**:理由 + 證據等級 + NCCN 條目。
- **第三層(深究,被質疑或懷疑時)**:原始變異、PRISMA 漏斗、原始文獻。

二三層永遠只差一次點擊,絕不強迫展開。不會寫程式的人活在第一層;深度是給「信任」用的,不是日常。`mtb-report.jsx` 實作了這個模型。

## 三個畫面

1. **工作清單(落地頁)** — 多位主治的共用收件匣。每列:病歷號、癌別、日期、負責醫師、狀態(處理中 / 待審閱 / 已審閱 / 已簽核)。可篩選、可分派。登入先看這個,不是直接進報告——它其實比報告更早決定採用。
2. **報告** — 表頭放病人 metadata bar(病歷號、診斷、分期、前線治療),下方分頁:摘要｜變異｜NCCN｜證據。
3. **證據 drill-down** — 點一個 variant 展開其註解、對應 NCCN 條目、支持文獻、迷你 PRISMA 漏斗。

## 信任層 = 設計本身(不是加分功能)

在腫瘤治療領域,這是骨架:

- **無裸述句**:每一句 AI 判讀都掛 inline citation chip,點開見來源。做不到來源的,視覺上與「有來源的事實」明確區分。
- **證據等級用顏色編碼,且顏色必須有意義**(不裝飾):第 I 級/可用藥/高證據 → 綠(success);第 II 級/中 → 琥珀(attention);意義未明/低 → 中性灰;抗藥性/禁忌 → 紅(danger)。畫布其餘保持 Primer 冷灰白,讓顏色只承載臨床意義。
- **簽核是一等公民、gated action**:醒目 primary、擋在「各段已審閱」之後、附署名 + 稽核紀錄、**可撤銷**。消除「怕弄壞」的凍結感。
- **常駐 Banner**:報告頂端常駐「AI 產生,需主治審閱後方生效」。

## Agent 透明度(也是最抓眼的一幕)

VCF 上傳後不要只轉圈。用 checks/Timeline 樣式把 pipeline 攤開:NGS 判讀 ✓、臨床脈絡擷取 ✓、NCCN 比對 ⟳、文獻合成 ⟳。既建立信任,又讓醫師知道系統在做什麼,demo 時是最抓眼的一幕。

## 門口:消滅打字與設定(最大採用槓桿)

醫師討厭打字、更不會設定參數。VCF + 病歷號兩欄的空白狀態就夠,價值全在下游。病歷號用條碼/刷手圈帶入。理想是**報告早就在工作清單裡等他**(檢驗系統交件時自動生成),而不是要他去找檔案、還得懂什麼是 VCF。門口的摩擦決定他有沒有機會看到價值。

## 反直覺:不要用聊天機器人當主入口

對忙、非技術的醫師,chatbot 反而抬高門檻(他得先知道要問什麼)。預先寫好、用他熟悉文體的結構化報告才是降門檻。「問這份報告」可以有,但當**次要功能**(ghost 按鈕),不搶主入口。

## 上手 = 預載示範病例 + 就地提示,不是說明書

專家不看手冊。讓他點一份現成報告 30 秒就懂;每個操作可撤銷,消除凍結感。

## 一個必守的可用性殺手

**意義未明 / 低證據的變異,預設必須收合、可降噪**——否則主治被雜訊淹沒,這是分子報告最常見的可用性殺手。

## 參考:Primer 元件與 token

可直接對應的 Primer 元件:Blankslate(空狀態)、DataTable(變異表/工作清單)、Label/Token(分級、狀態)、StateLabel(大狀態 pill)、Timeline(agent 執行/稽核軌跡)、Banner/Flash(警示)、UnderlineNav(報告分頁)、ActionList/NavList(側導覽)、PageHeader(報告表頭)、Dialog(簽核確認)。字體用系統 stack + 等寬(所有 VCF 來源資料)。
